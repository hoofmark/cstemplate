# cstemplate Example 4: NuGet Package References

This example uses NuGet packages `Dapper` and `Microsoft.Data.Sqlite` to create a local Sqlite database, populate it with some dummy data, and then generate simple reports. This is perhaps an unusual or extended use of templating, but shows how a template could for example be used by authenticated users to pull information from an existing database connection or web API and generate metadata, reports or scripts.

## Template Source Code: 

sample04.template.cs

### Template entry point and overall logic

This is the main body of the template. There are three main processing steps:
- Read configuration and initialise the database
- Populate the database with some sample data
- Query the database and generate reports

Extension methods are defined for the main functions, which keeps the high-level logic concise and self-explanatory.

```csharp
using HoofMark.CSharpTemplating.Abstractions;
using Dapper;
using System;
using System.Collections.Generic;
using System.Data;
using System.IO;
using System.Linq;
using System.Text;
using System.Threading.Tasks;
using Microsoft.Data.Sqlite;

namespace samples.sample04;

/// <summary>
/// A template that uses Dapper and Microsoft.Data.Sqlite to create a SQLite database, populate it with data and generate reports.
/// </summary>
public class Template04 : ITemplate
{
    public static void Run(ITemplateContext context)
    {
        // Read configuration values and initialize the database
        var dbPath = context.Config.Get<string>("dbPath") ?? "undefined.db";
        var tables = context.Config.Get<List<TableDefinition>>("tables") ?? [];
        using var connection = tables.InitDatabase(dbPath);

        //  Populate the database with sample data        
        connection.AddUser(1, "Alice", "alice@example.com");
        connection.AddUser(2, "Bob", "bob@example.com");
        connection.AddUser(3, "Charlie", "charlie@example.com");

        connection.AddProduct(1, "Laptop", 999.99m);
        connection.AddProduct(2, "Smartphone", 499.99m);
        connection.AddProduct(3, "Tablet", 299.99m);

        //  Generate a simple report of users and products
        var users = connection.ListUsers();
        var products = connection.ListProducts();   

        context.WriteFile("reports/users.txt", users);
        context.WriteFile("reports/products.txt", products);

    }
}
```

### Extension methods

1. Database Initialisation: context.WriteFile() only supports text file formats, so templates that read and output binary files or databases have to manage their own file paths / connection strings. Here, the SQLITE DB file is created in a temporary (APPDATA) location.

```csharp
public static IDbConnection InitDatabase(this IEnumerable<TableDefinition> tables, string dbPath)
{
    dbPath = Path.Combine(Path.GetTempPath(), dbPath);
    Directory.CreateDirectory(Path.GetDirectoryName(dbPath)!);

    // Delete the database file if it already exists, to start fresh
    if (File.Exists(dbPath))
    {
        File.Delete(dbPath);
    }

    // Create a new SQLite database file
    var connectionString = $"Data Source={dbPath};Mode=ReadWriteCreate;Cache=Shared";
    var connection = new SqliteConnection(connectionString);

    // Create the sample tables
    try 
    {
    connection.Open();
    foreach (var table in tables)
    {
        var createTableSql = table.TableCreateSql();
        connection.Execute(createTableSql);
    }
    }
    catch (Exception ex)
    {
        Console.WriteLine($"Error initializing database {dbPath}: {ex.Message}");
        throw;
    }

    return connection;
}
```

```csharp
public static void AddUser(this IDbConnection connection, int userId, string userName, string email)
{
    var sql = "INSERT INTO User (UserId, Username, Email) VALUES (@UserId, @Username, @Email)";
    connection.Execute(sql, new { UserId = userId, Username = userName, Email = email });
}

public static void AddProduct(this IDbConnection connection, int productId, string productName, decimal price)
{
    var sql = "INSERT INTO Product (ProductId, ProductName, Price) VALUES (@ProductId, @ProductName, @Price)";
    connection.Execute(sql, new { ProductId = productId, ProductName = productName, Price = price });
}
```

3. Querying the database and generating simple reports

Output at this point could of course be HTML, XML, SQL scripts, or any other text-based format. 

```csharp
public static string ListUsers(this IDbConnection connection)
{
    var users = connection.Query<User>("SELECT * FROM User").ToList();
    var sb = new StringBuilder();
    sb.AppendLine("Users:");
    foreach (var user in users)
    {
        sb.AppendLine($"  - {user.Username} ({user.Email})");
    }
    return sb.ToString();
}

public static string ListProducts(this IDbConnection connection)
{
    var products = connection.Query<Product>("SELECT * FROM Product").ToList();
    var sb = new StringBuilder();
    sb.AppendLine("Products:");
    foreach (var product in products)
    {
        sb.AppendLine($"  - {product.ProductName}: ${product.Price:F2}");
    }
    return sb.ToString();
}
```

4. Creating database tables using the metadata supplied in the config file (this is the same logic as sample03)

```csharp
/// <summary>
/// This template illustrates how to use a StringBuilder to generate SQL for table creation based on the table's columns.
/// </summary>
/// <param name="table"></param>
/// <returns></returns>
public static string TableCreateSql(this TableDefinition table)
{
    var sb = new StringBuilder();
    sb.AppendLine($"CREATE TABLE {table.Name} (");
    foreach (var column in table.Columns)
    {
        sb.AppendLine($"    {column.Name} {column.Type.MapCSharpTypeToSqlType()},");
    }
    sb.Remove(sb.Length - 3, 1); // Remove the last comma
    sb.AppendLine(");");
    return sb.ToString();
}
    
public static string MapCSharpTypeToSqlType(this string csharpType)
{
    return csharpType switch
    {
        "int" => "INT",
        "string" => "VARCHAR(255) NOT NULL",
        "string?" => "VARCHAR(255)",
        "bool" => "BIT",
        "DateTime" => "DATETIME",
        "decimal" => "DECIMAL(10,2)",
        _ => throw new ArgumentException($"Unsupported C# type: {csharpType}")
    };
}
```


## Template config

This uses the same metadata model as sample03. This simple model can easily be extended with concepts like primary and foreign keys, SQL datatypes, etc.

Rather than embedding the model in the template config, an alternative option would be to maintain the model in a separate `model.json` file, and record the (relative) path to that model definition file in the template config file. The template would then use `context.ReadFile(<relativePath>)` to retrieve the model. This is useful if several templates need to share a single model.
 
The config also includes a relative path for the (temporary) SQLITE database file. `cstemplate` places no restrictions on config file entries, so an absolute path would also be acceptable here.

sample04.template.json
```json
{
  "dbPath": "data/sample04.db",
  "namespace": "Sample04.Database",
  "tables": [
    {
        "name": "User",
        "columns": [
            { "name": "UserId", "type": "int" },
            { "name": "Username", "type": "string?" },
            { "name": "Email", "type": "string?" }
        ]
    },
    {
        "name": "Product",
        "columns": [
            { "name": "ProductId", "type": "int" },
            { "name": "ProductName", "type": "string?" },
            { "name": "Price", "type": "decimal" }
        ]
    }
  ]  
}
```

## Template context

For simplicity in these examples, the template context file is placed in the same directory as the template itself. In general, the context file can be placed anywhere in the folder hierarchy; `cstemplate` will scan parent folders and will use the first context file it finds in the folder tree.

This example shows how to include NuGet package names in the context file. These packages **MUST** also be referenced in the `.csproj` file (the *parent* .csproj which owns the *.template.cs). `cstemplate` uses the Nuget package information generated by `dotnet restore` to locate the dependent package versions required to compile this template.

cstemplate.config.json
```json
{
  "outputRoot": "../output/sample04",
  "project":    "../samples.csproj",
  "nugetPackages": [ "Microsoft.Data.Sqlite", "Dapper" ],
  "references": []
}
```

- `output root` identifies the path to the root folder, relative to the location of cstemplate.config.json
- `project` is the parent *.csproj file that "owns" the *.template.cs file
- `nugetPackages` lists referenced NuGet package names; also included as PackageReference entries in the parent samples.csproj file
- `references` lists local assembly references; not used in this example
